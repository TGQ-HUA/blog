## 引子：一个"null"让我查了四个小时

上个月车间日报那个宏跑的好好的，突然有一天罢工了。

就是那个我写了半天的自动汇总宏，打开WPS点一下按钮，三秒钟把昨天三个班组的产量从数据库拉出来填到日报模板里，然后发给老板。这东西用了一个多月从来没出过问题。

那天小周跑过来说："唐工，宏报错了，红字。"

我打开一看，控制台里一行：

```
TypeError: Cannot read properties of undefined (reading 'Value2')
```

哪个变量是undefined？不知道。哪一行？也没告诉是哪一行，WPS JS的报错有时候就一句，不给你堆栈。我当时就懵了。

我写了十几年Delphi，`.Value`怎么可能变`undefined`呢？Delphi里一个变量没赋值，编译期就报错了，不会让你跑到运行时再猜。

找了一个下午，最后发现是数据库里某个班组昨天没干活，没有产量记录。我那个代码直接取`Cells(1, 3).Value2`，单元格是空的，但WPS返回的不是`null`，是`undefined`。然后后面所有对这个值的操作都崩了。

这个事情让我意识到：从Delphi这种强类型、编译期查错的环境跳到一个动态类型、运行时才报错的JS宏里，有些思维惯性不改，迟早要栽。

## 正文：Delphi老狗到WPS JS，最疼的三个坑

### 坑一：undefined不是null不是空字符串，它什么都不是

在Delphi里，一个字符串变量要么有值要么是空串。你声明了 `var s: string;` 还没赋值，它默认是空串 `''`。调用 `Length(s)` 返回0，不会崩。

但WPS JS里，一个没赋值的变量是 `undefined`，不是`null`，不是`""`，是一个单独的类型。而且WPS的对象方法经常返回`undefined`而不是`null`。例如：

```javascript
// 取单元格的值
let val = ws.Cells(1, 1).Value2;
// 如果单元格是空的，val = undefined
// 如果你直接用 val.toString() 或 val.includes() —— 崩
```

在Delphi里我从来不检查字符串是不是`nil`，因为Delphi字符串不是对象。但JS里必须养成习惯：**每次从单元格读值，先判断是不是undefined**。

后来我的代码全改成了这样：

```javascript
function 安全取值(cell) {
    let v = cell.Value2;
    return (v === undefined || v === null) ? "" : v;
}
```

说起来简单，但踩这个坑之前我浪费了大半天。

### 坑二：WPS集合是1索引，JS数组是0索引——混在一起够你受的

Delphi里数组默认0索引（你写`array[0..N]`），但TStringList也是0索引。习惯了。

WPS里也一样，大部分集合从1开始：`Worksheets(1)` 是第一个工作表，`Rows(1)` 是第一行。这跟Delphi的TStringList[0]不一样，但不算难记。

难的是**当你在一个函数里同时操作WPS集合和JS数组，脑子会打架**。

比如我从数据库拉数据回来，存进一个JS数组 `let data = [];`，然后把它写进WPS表格里：

```javascript
// JS数组是0索引
for (let i = 0; i < data.length; i++) {
    // WPS单元格是1索引
    ws.Cells(i + 1, 1).Value2 = data[i]; // 记得+1
}
```

这么写没问题。但如果你哪次忘了+1……第一行数据写到第0行，WPS不报错，它默默写到了一个不存在的行前面，然后你就看到第一行是空的，数据从第二行开始。查半天才反应过来。

### 坑三：Range.Value2返回值类型不固定——一个函数两种行为

这是最阴的一个。

在Delphi里，你声明一个函数返回值是什么类型，它就一定返回什么类型。编译期就定死了。

但WPS的 `Range.Value2` 不一样：

- 如果你取**一个单元格**：返回一个**标量值**（数字或字符串）
- 如果你取**一片区域**：返回一个**二维数组**

```javascript
// 一个单元格
let v = ws.Cells(1, 1).Value2;
// v 可能是 "风管" 或者 123 或者 undefined

// 一片区域
let arr = ws.Range("A1:C10").Value2;
// arr 是一个二维数组，arr[0][0] 是第一行第一列的值
```

这本来不算大问题。问题是：**你事先不知道区域有多大**。比如你用 `UsedRange` 取数据，系统告诉你有多少行多少列，但你对返回值的处理方式跟取单格完全不同。

我第一次写总结日报宏的时候，先判断一下区域大小，如果只有一行就按单格处理，多行就按数组处理，加了个 `if (rows == 1 && cols == 1)` 的分支。这就是Delphi思维逼出来的防御式编程。不这样写，万一某天数据库只返回一条记录，宏就又崩了。

### 坑四：Application.Run传参数——说多了都是泪

这个坑我踩得最莫名其妙。

你要在WPS JS宏里调用另一个宏，用 `Application.Run("宏名", 参数1, 参数2)`。在VBA里这么写没毛病。

但在WPS JS里，如果你传的参数是**字符串**，而且字符串里包含中文——在某些WPS版本里它会变成乱码。我当时怀疑是不是我的代码有问题，后来发现是WPS的bug（或者说feature），跟WPS版本有关。同一个宏，在WPS 2021上正常，在WPS 2019上中文参数就变问号。

解决方案？不传字符串，传整数编号，让被调用的宏自己查字典。

这种"绕过去"的搞法，在Delphi里我从来没干过。Delphi的参数传递规规矩矩的，类型不对编译器就叫。

## 延伸：小厂技术人员的"自学困境"

我不是科班出身的程序员。我是做风管的，技术部主管，管排版、管下料、管车间。学Delphi是十年前白天干完活晚上回去自学，学WPS JS宏也是同一个模式，白天车间有问题，晚上回家搜资料。

但市面上WPS JS的资料真的不多。大部头书没有，网上文章东一篇西一篇，而且很多是写给VBA转JSA的程序员看的，对我这种Delphi转过来的"不对口"。

我经常会遇到一个问题，搜了半小时，发现唯一一篇相关的文章是官方文档里的，全英文，而且例子是VBA的。硬着头皮看，把VBA语法在脑子里翻译成JS。

那段时间我在Obsidian里建了一个笔记叫"WPS JS踩坑录"，每踩一个坑就记一条。到写这篇文章的今天，这个笔记有23条记录了，光"undefined"相关的就6条。

如果哪天有人整理一本《WPS JS宏从入门到放弃到真香》，我一定买一本。

## 有同行问我是怎么挺过来的

说几个我觉得管用的，不一定对：

| 问题 | 我在Delphi里怎么做的 | WPS JS里我的应对 |
|------|---------------------|-----------------|
| 取单元格值 | 直接取，类型固定 | 先判undefined，再转字符串 |
| 数组索引 | 统一0索引 | 看清楚是JS数组还是WPS集合，提前在注释里标 |
| 函数返回值 | 声明什么就是什么 | 不确定就加typeof检查 |
| 参数传递 | 编译期检查 | 能用数字就不用字符串 |
| 调试 | IDE断点 + 变量监视 | MsgBox + 写日志到单元格（土办法） |

不高级，但在这个行当里，好用的比好看的重要一万倍。

## 结尾

到现在我还在用Delphi写管理系统，用WPS JS宏做报表自动化。两套工具各干各的，谁也别想替代谁。但每次从Delphi切到JS宏，脑子里还是得先转个弯，"这里它不是Delphi，别用Delphi的习惯写JS"。

小周说我这个宏现在挺稳了，没再崩过。我说那是因为我先把那二十三个坑用代码填平了，不是因为我水平高。

老李当时在旁边来了一句："就跟咱做风管一样，不是手艺多好，是错犯够了。"
