## 引子：一个真实问题

去年六月份，我干了一件蠢事。

当时技术部的排版文件管理全靠共享文件夹，每人一个目录，命名全靠自觉。想查三个月前某个项目的排版，得挨个文件夹翻。

我跟老板说：我写个系统管起来。

老板说：你会写？

我说：现学。

然后我就真的去学了 Delphi。买了一本《Delphi基础教程》，看了三天第一章——"欢迎使用Delphi"——之后，直接打开 RAD Studio，开始写代码。

这不是励志故事的开头，这是灾难的开头。

## 正文：踩坑实录

### 一、第一个版本：能跑就是胜利

第一个版本我写了大概两个月。功能很简单：一个登录界面、一个主窗口、几个数据录入表单、一个 MySQL 数据库。

界面风格是"Delphi 默认主题"——灰色面板、凸起按钮、3D 边框，看着像 Windows 98 时代的古董。但我当时不觉得丑啊，我觉得"能连上数据库了！牛逼！"

这个版本最大的问题是：**所有代码都写在一个 Unit 里**。

按钮点击事件、数据库查询、窗口切换、报表生成……两千多行代码，全在一个 .pas 文件里。变量命名是 `Edit1`、`Edit2`、`Button1`——Delphi 默认拖控件生成的，我一个都没改。

你要是现在打开那个版本，你绝对看不懂哪段代码是干嘛的。我自己也看不懂了。写的时候觉得"这不是很清楚吗"，三个月后打开——这他妈谁写的？

是我写的。

### 二、第一个功能上线：报价排版留档

不管代码多烂，老板要的是"能用"。

第一个真正上线的功能是"报价排版留档"。技术员在 CAMduct 排好版，打开系统，点一下按钮，文件自动复制到部门共享盘和本地存档，同时写入数据库记录。

流程很简单，但对我这个新手来说，每个环节都是坑：

1. **路径问题**——共享盘符号在每台电脑上不一样，有的电脑是 Z 盘，有的是 Y 盘。我写死在代码里，结果换台电脑就崩。后来才学会读配置文件
2. **文件复制冲突**——两个人同时点"留档"，文件互相覆盖。我加了时间戳后缀，但又忘了判断同名文件
3. **数据库连接池**——一开始每次操作都新建连接，用完不关。跑了一天，MySQL 报"Too many connections"。车间打电话来说系统崩了，我在家里饭都没吃完跑回厂里修
4. **中文乱码**——Delphi 连 MySQL，中文全变问号。百度了两天才搞定 UTF-8 编码设置

上线第一周，我修了大概 15 个 Bug。每天下班回家，手机一响就紧张——不是车间打来的，就是技术部打来的。

### 三、第二个月开始重构

扛了一个月，Bug 少了，新需求来了。

生产主管说：能不能加个下单排版的功能？
跟单说：能不能查这个月的报价工作量？
老板说：能不能统计每个人的绩效？

如果是第一个版本的那个"两千行一个 Unit"的代码结构，加这些功能等于重新写一遍。我终于意识到一个问题：我写的这个"系统"，只有我能维护。因为代码只有我看得懂——不对，是连我自己都快看不懂了。

于是我开始第二次重写：

- 把功能拆成独立的 Unit（登录、主界面、报价排版、下单排版、人员管理……）
- 建了一个公共数据库模块，所有连接都走它
- 配置文件外置，不写死路径
- 变量终于敢用中文命名了，Delphi 是支持 Unicode 的，`FProjectName: String;` 至少一看就懂

同时我开始用 Git 做版本管理。不是因为我懂，是因为有一次改崩了代码，回退花了三个小时。那天我在 obsidian 笔记里写了一句话："不改版本管理系统的人，迟早被自己的代码搞死。"

### 四、现在的版本

现在系统跑了一年了，14 个功能模块，20 多个辅助工具，车间每天在用。最新的版本号是 V1.0——从 V0.1 到 V0.9，中间重构了三次。

写代码这块，我到现在都不算"会"。不会设计模式，不会面向对象，多线程搞不明白，遇到复杂的 SQL 还得百度。但我发现一件事：**对于工厂内部工具来说，60 分的代码，只要有人用、有人改、有人维护，就比 100 分的代码没人用强一百倍**。

这套系统最让我骄傲的不是代码写得多漂亮，是车间那个五十多岁的切割师傅，每天早上打开系统看一眼当天的排版任务。

## 延伸：行业思考

风管行业搞数字化，听上去很高大上。但实际上，大部分厂的"数字化"就是：
- 花几万块买个 ERP → 用不起来 → 弃了
- 找外包写个管理系统 → 沟通成本比开发成本还高 → 烂尾
- 自己人写 → 写得烂但能用 → 缝缝补补又三年

我选了第三条路。不是因为它好，是因为只有这条路走得通。

自己做工具的好处是：想要什么功能，今天想，明天就能加。出 Bug 了，自己半小时修好，不用等外包排期。坏处是：代码丑、有坑、只有自己知道怎么跑。

但随着时间推移，我发现另一个好处——**你做的工具会反过来塑造你的工作方式**。因为你知道能做成什么样，所以你会主动去想："这个流程能不能系统化？"而不是等着别人告诉你该做什么。

## 实操：想自己写工具的几点建议

如果你也想自己搞个内部工具，几条实在话：

1. **别追求完美架构，先把功能跑通**——第一个版本可以丑，但不能慢
2. **尽早拆分代码**——一个文件超过 500 行就该拆了，别犹豫
3. **版本管理从第一天开始**——Git 不熟没关系，`git init` 就一句话
4. **中文命名不丢人**——Delphi、Python、Java 都支持，`F客户名称` 比 `cust_name` 好维护
5. **配置文件外置**——IP 地址、路径、密码写死在代码里，换环境就是灾难
6. **留好错误日志**——出了 Bug 没日志等于瞎猜，`try..except` 里记一条日志比什么都管用
7. **接受你的代码会被嫌弃**——没关系，能用就行。等你写到第三版，自然就写好了

## 结尾：接地气的收束

上周有个新来的同事看到系统登录界面，问：这软件谁做的？

我说我做的。

他看了两眼，说了句：界面有点复古。

我说：能用就行。

其实心里想的是：一年前我连 Delphi 是什么都不知道，现在我的系统在车间跑了三百天没出过大故障。界面丑怎么了，它好用。

现在回头想，当初要是说"学一下"而不是"现学"，可能少走三个月弯路。不过话说回来，不先跌这一跤，也不知道坑在哪儿。
